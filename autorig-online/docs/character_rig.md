# Character rig: что происходит в таске (текущая реализация)

Документ описывает **фактическое поведение** бэкенда и интеграции с воркерами по состоянию кодовой базы `autorig-online`. Его можно использовать как основу для обсуждения фич (например, экспорт спрайт-листов из вьювера).

---

## 1. Что такое «таск» в продукте

Пользователь передаёт **URL 3D-модели** (или загружает файл — тогда сервис кладёт файл в `UPLOAD_DIR` и отдаёт публичный URL вида `{APP_URL}/u/{token}/{filename}`). Создаётся запись `Task` в SQLite с полями владельца (`anon` / `user`), `input_url`, `input_type` (режим рига, по умолчанию `t_pose`), статусом `created`.

Дальше таск **ставится в очередь** и уходит на **внешний воркер** (конвертер AutoRig), который выполняет риг и выкладывает набор выходных файлов по HTTP. Сайт **не считает геометрию и не ригует сам** — только оркестрация, прогресс, кэш, биллинг скачиваний и UI.

---

## 2. Создание таска (`POST /api/task/create`)

- **Тело:** JSON или form; поля `input_url`, `type` / `input_type` (пусто → `t_pose`), опционально `ga_client_id`, загрузка файла при `source=upload` + `file`.
- **Нормализация типа:** `normalize_task_type()` — пустая строка становится `t_pose`.
- **Запись в БД:** `create_conversion_task()` — статус `created`.
- **Немедленная отправка:** если `get_global_queue_status()` находит **свободного воркера** (доступен, не перегружен, очередь 0), вызывается `start_task_on_worker()` для этого таска. Иначе таск ждёт **фоновый цикл** (`background_task_updater`), который периодически разбирает очередь `status=created`.

Списание кредитов за **сам факт создания** таска в `api_create_task` **не реализовано** (функции `decrement_user_credits` / лимиты anon в `auth.py` есть, но в потоке создания таска не вызываются). Кредиты используются в **покупках** (файлы пакета, отдельные кастомные анимации) — см. раздел 8.

---

## 3. Вызов воркера

`send_task_to_worker(worker_url, input_url, task_type)`:

- **HTTP:** `POST` на URL воркера (в конфиге это `.../api-converter-glb`, плюс таблица `worker_endpoints` с весами).
- **JSON:** `input_url`, `type` (как `input_type` таска), **`mode`: `"only_rig"`** — отдельные режимы конвертации на стороне сайта не задаются, только риг.
- Опционально при **рестарте** таска: `local_position` / `local_rotation` / `local_scale` из тела запроса или из сохранённых `viewer_settings.modelTransform`.

**Ответ воркера (успех):** `task_id`, список **`output_urls`** (ожидаемые файлы), `progress_page` / `progress_url`, из которых извлекается **`guid`** (UUID задачи на стороне воркера).

Поддерживаемые входы на стороне воркера (как задокументировано в коде): **GLB, FBX, OBJ** по одному `input_url`.

### FBX → GLB (отдельный эндпоинт)

В `workers.py` есть `send_fbx_to_glb()` — вызов `{worker_base}/api-converter-glb-to-fbx` с `input_url` на FBX. Асинхронная обвязка `_start_fbx_preconvert_async()` в `tasks.py` **нигде не вызывается**; основной путь — **прямая** отправка исходного `input_url` в `send_task_to_worker`. Поля `fbx_glb_*` в модели `Task` и флаг `prepared_glb_ready` (в т.ч. для прокси prepared GLB) рассчитаны на сценарий с предконвертацией, но **живой пайплайн создания таска** идёт без неё.

## 4. Прогресс и завершение

- Пока `status == processing`, фоновый воркер вызывает `update_task_progress()`:
  - по батчам проверяется **HEAD** по URL из `output_urls`; появившиеся попадают в `ready_urls`, растёт `ready_count`;
  - при `ready_count >= total_count` → `status = done`;
  - отдельно проверяется **видео:** `{worker_base}/converter/glb/{guid}/{guid}_video.mp4` — при 200 выставляются `video_ready`, `video_url`.
- При переходе в `done`: опционально email владельцу, Telegram, GA4 `rig_completed`, **кэширование файлов** `cache_task_files()` в каталог статики таска (`TASK_CACHE_DIR / task_id`), выдача потом через `/api/file/...`.

**Зависшие таски:** `find_and_reset_stale_tasks()` — глобальный таймаут и «нет прогресса» N минут; сброс в `created` для повторной постановки или `error` после лимита рестартов.

---

## 5. Что лежит в `output_urls` / `ready_urls` (логика сайта)

Сайт **не парсит** внутренности пайплайна воркера; он знает только список URL. По соглашению с воркером в путях фигурируют **папки качества** `_100k`, `_10k`, `_1k`.

Для API статуса таска вычисляются:

- **`viewer_html_url`** — первый `.html` в `ready_urls` с `_100k/` (см. `find_file_by_pattern`).
- **`quick_downloads`** — по тем же правилам: `.max`, `.ma`, `.c4d`, `.hdrp.unitypackage`, `.unitypackage`, `.fbx` в `_100k` (приоритетные ярлыки для DCC/движков).
- **`prepared_glb_ready`** — есть `_model_prepared.glb` среди готовых URL **или** (для FBX-сценария с предконвертацией) `fbx_glb_ready`.

Публичный просмотр прогресса воркера: URL вида  
`{worker_base}/converter/glb/{guid}/{guid}.html`  
и лог `{guid}_progress.txt` (эндпоинт `/api/task/{id}/progress_log`).

## 6. Кастомные анимации (каталог, не «магия вьювера»)

- Статический каталог: `static/all_animations/manifest.json` + метаданные и превью по файлам в `static/all_animations/`.
- Для таска строится карта файлов анимаций **`_build_task_animation_file_map()`**:
  1. разбор URL из `output_urls` / `ready_urls` (отдельные `.fbx`, не `*_all_animations*`);
  2. при необходимости — запрос к воркеру **`GET /api-converter-glb/model-files/{guid}`** и сбор `.fbx`;
  3. если список с воркера пуст, но в выдаче есть пакет вида `*_all_animations_unity.fbx` и известен manifest — **синтетические URL** по шаблону `{worker_root}/{guid}/{guid}_{anim}.fbx` (помечаются как synthetic).
- API: `/api/task/{id}/animations/catalog`, покупка отдельных анимаций или бандла, выдача/прокси скачиваний (в т.ч. стрим для превью в вьювере). Цены в коде: **1** кредит за анимацию, **10** за все (константы в `main.py`).

Итого: **набор анимаций** для персонажа определяется **воркером** (генерация FBX) + **манифестом** на сайте; фронт показывает каталог и привязку к файлам таска.

---

## 7. Дополнительно: face rig, настройки вьювера

- В `Task` хранятся **`viewer_settings`** (JSON) — сохранение состояния вьювера по таску.
- **`face_rig_analysis`** — опциональный анализ/кэш для face-rig UI (отдельные эндпоинты), **не** часть основного пайплайна рига на воркере.

---

## 8. Сводка: от загрузки до «все анимации»

| Этап | Где в коде / что |
|------|------------------|
| Создание записи таска | `create_conversion_task`, статус `created` |
| Выбор воркера | `select_best_worker`, очередь + фоновый цикл |
| Старт рига | `start_task_on_worker` → `send_task_to_worker` (`mode: only_rig`) |
| Список выходных файлов | из ответа воркера → `output_urls`, `total_count` |
| Готовность файлов | `check_urls_batch` (HEAD) |
| Видео превью | `check_video_availability` |
| Завершение | `status: done`, кэш файлов, уведомления |
| Список кастомных анимаций для UI/покупок | manifest + `_build_task_animation_file_map` + API покупок |

---

## 9. Замечание для следующей фичи (спрайт-листы)

Сейчас **нет** серверного рендера покадровой 2D-текстуры из сцены: всё 3D — в браузерном вьювере и в выгружаемых GLB/FBX/pакетах движков. Любая генерация пиксельного спрайт-листа поверх текущего стека потребует отдельного контура (клиентский offscreen render, серверный рендер, или новый этап на воркере) и явной спецификации: FPS, ракурс, разрешение, стилизация.

---
